Blueprint İleri — Profesyonel Mimari

Bölüm 4-6'da Blueprint'in temellerini öğrendin. Bu bölüm seni amatörden profesyonele taşır: Interface, Event Dispatcher, Inheritance, Soft Reference, Behavior Tree, Component Pattern — sürdürülebilir, ölçeklenebilir Blueprint sisteminin yapı taşları.

25.1

Neden ileri Blueprint? — Spagetti tuzağı

Yeni başlarken her şey tek bir Blueprint'in içine sığar — bir karaktere 50 fonksiyon eklersin, her objeyi Cast to BP_Player ile bağlarsın, oluyor. Sonra proje büyür: 20 düşman türü, 30 silah, 100 etkileşim... Cast zinciri kopar, "BP'yi açınca crash atıyor", "asset audit alınamıyor". Bu, spagetti Blueprint tuzağıdır.

Gerçek hayat benzetmesi

Bir evin elektrik tesisatı düşün: Tek bir kablo doğrudan her cihaza gitseydi, bir cihaz değişince tüm sistemi yeniden çekmen gerekirdi. Bunun yerine sigorta panosu ve standart prizler var — cihaz cihaza takılır, biri bozulunca diğerleri etkilenmez. Bu bölüm Blueprint için "sigorta panosu + standart priz" mimarisini öğretiyor.

İleri Blueprint'in 6 ana aracı

Interface

"Bu objeyle konuşabilirsin" sözleşmesi. Cast olmadan iletişim.

Event Dispatcher

"Yayınlıyorum, dileyen dinler". Çok-dinleyicili olay sistemi.

Inheritance

Parent/Child mantığı. Ortak davranışı bir yerde tut, üzerine kişiselleştir.

Soft Reference

Hard yerine "gerektiğinde yükle" referans. Memory ve start-up kazancı.

Behavior Tree

AI'nın "ne yapacağına" karar veren ağaç yapısı.

Component Pattern

Yeniden kullanılabilir davranış parçaları. Kompozisyon > Kalıtım.

25.2

Interface — Cast'siz iletişim

Bir oyunda 20 farklı objeye "etkileşilebilir" olmasını istiyorsun: kapı, sandık, NPC, anahtar, mum... Klasik yol: her birinde "Interact" fonksiyonu var, oyuncu objeye baktığında Cast to Kapı, Cast to Sandık, Cast to NPC... diye 20 dallı bir zincir. Korkunç.

Çözüm: Blueprint Interface (BPI). "Bana Interact fonksiyonu olduğunu söz veriyorum" diyen bir sözleşme. Sonra oyuncu sadece "objeye Interact gönder" der — kim implement etmişse o cevap verir.

Gerçek hayat benzetmesi

Bir TV uzaktan kumandası farklı markaları çevirebilir. Kumanda "Açma sinyali" gönderir — TV, fırın, klima... hepsi farklı şirketlerin ürünü, ama hepsi "açılma" standardını implement eder. Kumanda hangi marka olduğunu bilmek zorunda değil; sinyali gönderir, anlayan cevap verir. Interface tam bunu yapar.

Interface oluşturma ve kullanma

1
Interface asset'i

Content Browser → sağ tık → Blueprints → Blueprint Interface. İsim: BPI_Interactable. Çift tıkla, içinde fonksiyon ekle: Interact (input parametre: Instigator, output: bool Success).

2
Interface'i implement et

Kapı Blueprint'ini aç → Class Settings → Implemented Interfaces → Add → BPI_Interactable. Artık My Blueprint panelinde "Interact" fonksiyonu görünür — çift tıkla, kapıya özel kodu yaz (kapıyı aç).

3
Sandık, NPC, vs için aynı

Sandık, NPC, Mum Blueprint'lerinde aynı interface'i implement et. Her biri kendi Interact mantığını yazar.

4
Karakterden çağır

Karakter Blueprint'inde: Line Trace → Hit Actor referans. Sağ tık → "Interact (Message)" ara — interface fonksiyonları "Message" suffix ile görünür. Cast yok. Sadece çağırırsın; implement etmiş olan cevap verir, etmemiş ise sessiz geçer.

"Call" vs "Message" farkı

Bir interface fonksiyonunu çağırırken iki seçenek vardır: Call (target interface'i implement etmeli, etmiyorsa hata) ve Message (etmemişse sessiz geçer). Çok obje türüne aynı şeyi göndereceksen Message kullan — daha güvenli.

İleri seviye — Does Implement Interface

Bazen önceden "bu obje interface'i taşıyor mu?" sorusunu sormak istersin. Does Implement Interface node'u true/false döner. Etkileşim UI prompt'unu sadece interaktif objelerde göstermek için ideal: oyuncu baktığında "Does Implement Interface BPI_Interactable" → evetse "E ile etkileşim" UI'ı gösterir.

25.3

Event Dispatcher — Çok dinleyicili sinyal

Interface "bir objeye iletişim kurmanın" yoluydu. Event Dispatcher ise bir objeden çoklu dinleyicilere sinyal yayma yoludur. "Boss öldü" sinyali çıkar; HUD haberleşir (skor güncelle), Quest Manager (görev tamamla), Audio Manager (zafer müziği) ayrı ayrı dinler.

Gerçek hayat benzetmesi

Event Dispatcher bir radyo istasyonudur. İstasyon yayın yapar — kim dinlemek isterse "abone olur" (Bind). Kimin dinlediğini istasyon bilmez, umursamaz. Yayın çıkar, dinleyiciler tepki verir. Yeni dinleyici eklemek için kaynağa dokunmazsın — sadece Bind eder.

Kullanım — Boss ölüm sinyali

1
Dispatcher tanımla

BP_Boss → sol panel → Event Dispatchers → +. İsim: OnBossDefeated. İstersen parametre ekle (örn. bossLevel: int).

2
Sinyali fırlat (Call)

BP_Boss → Event Graph → Health 0 olduğunda: Call OnBossDefeated node'u. Parametreleri doldur. Yayın bitti.

3
Dinleyiciler bind eder

HUD Widget'ta BeginPlay'de: Boss referansı al → Bind Event to OnBossDefeated → bir custom event'i hedef olarak ver. Custom event tetiklendiğinde skor güncellenir. Aynı şekilde QuestManager kendi event'ini bind eder.

Interface vs Event Dispatcher — Hangisi ne zaman?

Interface: "Bir hedefe bir şey söylemek". 1 → 1 (veya az). Yön: dışarıdan içeri.
Event Dispatcher: "Bir şey oldu, dileyen duysun". 1 → N. Yön: içeriden dışarıya.
Bir kapıyı açmak → Interface. Boss öldü, herkes haber alsın → Event Dispatcher.

25.4

Inheritance — Parent / Child Blueprint

5 farklı düşman türü yapıyorsun: Goblin, Ork, Şövalye, Büyücü, Köpek. Hepsinde health, damage, takeDamage, die ortak. 5 Blueprint'e aynı kodu kopyala-yapıştır mı yapacaksın? Hayır — Parent Blueprint yazarsın, diğerleri ondan miras (inherit) alır.

1
Parent oluştur

Yeni Blueprint Class → Parent Class: Character. İsim: BP_EnemyBase. İçine ortak variable (Health, Damage, AttackRange) ve ortak fonksiyon (TakeDamage, Die) yaz.

2
Child Blueprint üret

BP_EnemyBase'e sağ tık → "Create Child Blueprint Class". İsim: BP_Goblin. Açıldığında parent'taki tüm değişken ve fonksiyon hazır. Sadece farklılıkları (mesh, ses, özel saldırı) yaz.

3
Override (üzerine yaz)

Parent'taki bir fonksiyonu değiştirmek istersen My Blueprint paneli → fonksiyona sağ tık → Override Function. Yeni davranış yazarsın, parent'taki kullanılmaz olur. Parent: TakeDamage node'u ile yine de orijinal davranışı çağırabilirsin (super gibi).

Çok derin inheritance kötüdür

2-3 katman inheritance iyi, ama 5+ katman "diamond problem" ve karmaşa yaratır. Çok derinleşince Component Pattern'a geç (aşağıda). Kural: "Is-a" → Inheritance, "Has-a" → Component.

25.5

Soft Reference — Akıllı yükleme

Bir Blueprint başka bir asset'e Hard Reference verdiğinde, ilki yüklendiğinde diğeri de zorla RAM'e yüklenir. 100 farklı silahı destekleyen karakter, hepsini hard reference eden bir liste tutuyorsa — oyuncu sadece 1 silahı kullansa bile, 100'ü RAM'de. Bu, level başlangıç süresi ve memory patlamasının ana sebebidir.

Soft Reference: "Bu asset'i göstereceğim, ama şimdi yükleme, gerektiğinde yükle." Talep edildiğinde manuel Async Load Asset ile yüklersin.

Pratik tarif — Async load

1
Variable Soft Reference olarak ayarla

Karakterde EquippedWeapon variable. Tip: Soft Class Reference → BP_Weapon. Bu sayede karakter yüklenirken silah hemen RAM'e gelmez.

2
İhtiyaç anında yükle

Silahı kuşanma anında: Async Load Class Asset node → On Completed → spawn et. Async olduğu için frame'i bloklamaz.

İleri seviye — Size Map'i incele

Content Browser'da bir asset'e sağ tık → "Size Map". Hard reference zinciri görsel olarak çıkar. Şok edici sonuçlar görebilirsin: ufak görünen bir UI Widget'ın 200 MB asset'i tetiklediği keşifleri burada bulursun. Memory optimizasyonun başlangıç noktası.

25.6

Behavior Tree — AI karar ağacı

Düşman AI için "Tick içinde uzun if-else" yazmak hızla cehenneme döner. UE5'in Behavior Tree sistemi AI'nin "ne yapacağına" görsel bir ağaç ile karar verir. Solda yüksek öncelikli, sağda düşük öncelikli davranışlar.

Gerçek hayat benzetmesi

Bir bekçinin önceliklerini düşün: 1) Yaralanmışsa kaç. 2) Düşman görüyorsa saldır. 3) Devriye yap. Yukarıdan aşağı kontrol eder; yaralı değilse 2'ye, düşman yoksa 3'e. Bekçi ağacı bu sıra. Behavior Tree de aynı şekilde sol→sağ, üst→alt önceliklerle çalışır.

4 ana asset

Behavior Tree (BT)

Ağaç yapısı — root'tan node'lara doğru kararlar. BT_Enemy.

Blackboard

AI'nin "hafızası". Variable'lar (TargetPlayer, IsAlerted, LastKnownPos). BT bunları okur/yazar.

AI Controller

Bir Pawn'ı kontrol eden özel Controller. BT'yi Run Behavior Tree ile başlatır.

AI Perception

Görme/duyma — Sight Config, Hearing Config. Algıladığında Blackboard'a "TargetPlayer" yazar.

Behavior Tree örneği Selector + Sequence

Selector vs Sequence — İki ana composite

  • Selector (?): Çocukları sırayla dener, biri başarılı olunca durur. "Şunlardan biri olsun yeter" mantığı.
  • Sequence (→): Çocukları sırayla işler, biri başarısız olunca durur. "Hepsi sırayla yapılmalı" mantığı.
İleri seviye — Decorator & Service

Decorator'lar node'a bir koşul ekler (örn. "TargetPlayer != none ise bu dal çalışsın"). Service'ler arka planda belirli aralıklarla çalışır (örn. her 0.5 sn Blackboard'a player mesafesini yaz). İkisini birlikte kullanmak BT'yi son derece zekileştirir. Selector + Decorator + Service üçlüsü AAA AI'larının omurgasıdır.

25.7

Component Pattern — Kompozisyon kalıtıma karşı

Bir karakter hem yüzebilir hem tırmanabilir hem büyü yapabilir. Bunları tek bir Blueprint'e koymak şişman bir BP üretir. Inheritance ile çözmek istersen: BP_Char → BP_SwimmingChar → BP_ClimbingSwimmingChar → BP_MagicClimbingSwimmingChar... aşağıya doğru saçma uzar.

Çözüm: Davranışları ActorComponent'lere parçala. BP_SwimComponent, BP_ClimbComponent, BP_MagicComponent. Karakter ihtiyacı olan component'leri "takar". Has-a, is-a değil.

Gerçek hayat benzetmesi

Bir telefon düşün: yeni kamera, yeni hoparlör, yeni pil... Apple her özellik için yeni telefon modeli yapmaz; modülleri değiştirir. "Yüzme komponenti", "tırmanma komponenti" de aynı: bir karaktere takar, başka karakterden çıkarırsın. Component = takılabilir özellik modülü.

Custom Actor Component oluşturma

1
Component asset'i

Yeni Blueprint Class → Parent: ActorComponent. İsim: BPC_HealthComponent. İçine Health variable, TakeDamage fonksiyonu, OnDeath event dispatcher ekle.

2
Karaktere tak

BP_PlayerChar açıldığında: Add Component → BPC_HealthComponent. Karakter artık bir health'e sahip — bütün mantık component'ten geliyor.

3
Aynı component'i düşmana, NPC'ye, hatta kapıya tak

Düşman da, kapı da (kırılır kapı) aynı HealthComponent'i kullanabilir. Tek bir health mantığı, sınırsız obje. DRY (Don't Repeat Yourself) kuralının ta kendisi.

"Has-a" mantığı

Sorudur: "Karakterin health'i var (has-a) mı, yoksa karakter health türü mü (is-a)?" Cevap: Has-a. O zaman Component. "Goblin bir düşmandır" = Is-a → Inheritance. Doğru aracı bu basit testten seç.

Hap Bilgiler

  • Interface: Cast'siz çoklu obje türlerine konuşmak. 1→1.
  • Event Dispatcher: "Yayınlıyorum, dileyen dinler". 1→N.
  • Interface "Call" hatayı atar, "Message" sessiz geçer — güvenli olan Message.
  • Inheritance: "Is-a" — Goblin bir düşmandır.
  • Component: "Has-a" — karakter bir health'e sahiptir.
  • Override Function ile parent davranışı child'da değiştirilir.
  • Hard Reference: otomatik yükleme, yüksek memory.
  • Soft Reference: talep-based yükleme, düşük memory; Async Load ile çek.
  • Size Map ile gizli hard reference zincirlerini gör.
  • Behavior Tree: AI karar ağacı. Sol → Sağ öncelik.
  • Selector (?): biri başarılı olunca dur. Sequence (→): biri başarısız olunca dur.
  • Blackboard: AI hafızası. Variable'lar burada.
  • Kompozisyon kalıtıma yeğdir — derin inheritance'a girmeden component'e geç.

Pratik Projeler

İleri Blueprint mimarisini uygula
Kolay~30 dk
Proje 1: BPI_Interactable ile etkileşim sistemi

BPI_Interactable Interface oluştur (Interact fonksiyonu). 3 farklı obje yap: BP_Door (kapı açar), BP_Chest (sandık ışıldatır), BP_Lamp (lambayı açar/kapar). Hepsi BPI_Interactable implement etsin. Karakter Line Trace ile baktığı objeye Interact (Message) gönderir, hangi obje olduğunu Cast etmeden. E tuşunda çalışsın.

Tamamlama Kontrolü
  • BPI_Interactable interface'i oluşturuldu
  • 3 obje hepsinde implemented
  • Karakter Cast olmadan iletişim kuruyor
  • Her obje farklı tepki veriyor
  • Hiç Cast node'u yok
Cast cehenneminden kurtulup interface ile temiz iletişim kurmak.
Orta~50 dk
Proje 2: HealthComponent + Event Dispatcher

BPC_HealthComponent Actor Component oluştur: Health (float), TakeDamage(amount), OnDeath Event Dispatcher. Karaktere ve 2 düşman türüne (BP_Goblin, BP_Ork) ekle. HUD Widget'ta, karakterin HealthComponent'inin OnDeath'ine bind ol — GameOver UI göster. Bir QuestManager Actor'da, düşmanların OnDeath'ine bind olup öldürülen sayıyı tut.

Tamamlama Kontrolü
  • BPC_HealthComponent oluşturuldu (variable + fonksiyon + dispatcher)
  • Karakter ve 2 düşman türünde component takılı
  • HUD karakterin OnDeath'ine bind
  • QuestManager düşman OnDeath'ine bind
  • Component pattern: Health mantığı tek bir yerde
Component + Event Dispatcher ikilisini gerçek senaryoda kullanmak.
İleri~90 dk
Proje 3: AI Düşmanı — Behavior Tree + Inheritance + Component

BP_EnemyBase parent class: HealthComponent + AI Controller + Behavior Tree. BP_Goblin ve BP_Ork child class'ları, sadece mesh/saldırı farklı. Behavior Tree: Selector → (Sequence: Saldır), (Sequence: Devriye). Blackboard'da TargetPlayer. AI Perception ile karakteri görünce TargetPlayer yazılır. BTTask'lar: MoveToTarget, Attack, FindWaypoint, MoveToWaypoint.

Tamamlama Kontrolü
  • BP_EnemyBase parent oluşturuldu
  • 2 child (Goblin, Ork) miras alıyor, farklılaşıyor
  • HealthComponent component pattern olarak takılı
  • BT_Enemy + BB_Enemy oluşturuldu
  • AI Perception → Blackboard'a TargetPlayer yazıyor
  • Selector ile saldır/devriye geçişi çalışıyor
  • BT debug görünür (üst menü → Show Behavior Tree)
4 ileri sistemin birlikte çalıştığı tam profesyonel AI'yi inşa etmek.
